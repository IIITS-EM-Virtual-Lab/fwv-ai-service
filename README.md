# FWV AI Service — Fieldora

AI-powered tutoring chatbot for the **Fields and Waves Visualisation Lab (FWV Lab)** at IIIT Sri City.
Built on RAG (Retrieval-Augmented Generation) using FAISS + Llama-3.3-70b via Groq, with Gemini as fallback.

## Quick Start — Run Locally

### 1. Clone the repo

```bash
git clone https://github.com/IIITS-EM-Virtual-Lab/fwv-ai-service.git
cd fwv-ai-service
```

### 2. Create virtual environment

```bash
python -m venv .venv

# Windows
.venv\Scripts\activate

# Mac/Linux
source .venv/bin/activate
```

### 3. Install dependencies

```bash
pip install -r backend/requirements.txt
```

### 4. Set up environment variables

Create `backend/.env`:

```env
# Groq API Keys (free at console.groq.com)
GROQ_API_KEY_1=gsk_xxxxxxxxxxxxxx
GROQ_API_KEY_2=gsk_xxxxxxxxxxxxxx

# Gemini API Keys (free at aistudio.google.com)
GEMINI_API_KEY_1=AIxxxxxxxxxxxxxx
GEMINI_API_KEY_2=AIxxxxxxxxxxxxxx

# FWV Lab site URL (for embedded links)
FWV_SITE_URL=https://www.fwvlab.com
```

### 5. Build the FAISS index

Run this once before starting the server:

```bash
cd backend
python build_index.py
```

This generates `backend/index/faiss.index` and `backend/index/metadata.pkl`.

### 6. Start the backend server

```bash
uvicorn backend.app:app --reload
```

Server runs at: `http://127.0.0.1:8000`

### 7. Open the frontend

Open `frontend/index.html` directly in your browser.

## API Reference

### `POST /ask`

Ask a question to Fieldora.

**Request:**
```json
{
  "query": "What is Faraday's Law?",
  "session_id": "user-abc123"
}
```

**Response:**
```json
{
  "answer": "Faraday's Law states that a changing magnetic field induces an electric field.\n\nTo learn more, visit: <a href='...'>Faraday Law</a>\n\nArea: Maxwell Equations\nTopic: Faraday Law"
}
```

## How It Works

```
User Question
     ↓
Frontend (script.js)
     ↓ POST /ask with session_id
FastAPI (app.py)
     ↓
RAG Service (rag_service.py)
  - Encodes question using sentence-transformers
  - Searches FAISS index for top 5 relevant chunks
     ↓
LLM Service (llm_service.py)
  - Builds prompt with context + session memory
  - Tries Groq Key 1 → Groq Key 2 → Gemini Key 1 → Gemini Key 2
  - Appends topic link and area label
     ↓
Response sent back to frontend
```

## Dataset Generation

To regenerate or extend the training dataset:

```bash
python backend/generate_dataset.py
```

- Uses Groq (Llama-3.3-70b) to generate QA pairs
- Covers all FWV topics from Sadiku's *Elements of Electromagnetics*
- Saves progress to `checkpoint.jsonl` after every subtopic
- Output: `fwv_dataset.jsonl` (~676 pairs)

**Note:** Groq free tier allows 14,400 requests/day.
If the script stops due to rate limits, just run it again the next day —
it automatically resumes from the checkpoint.

## Session Management

Each user gets a unique session ID generated in the browser:

```javascript
// Generated once per browser, stored in localStorage
const sessionId = crypto.randomUUID();
localStorage.setItem("fwv_session_id", sessionId);
```

Logged-in users use their account ID: `user-{mongodb_id}`
Guest users use: `anon-{uuid}`

This ensures each user has isolated conversation memory.

## Fallback Chain

If one AI service is down or rate limited, the system automatically tries the next:

```
Groq Key 1 → Groq Key 2 → Gemini Key 1 → Gemini Key 2 → Error message
```

## Notes

- Never commit `.env` to git — it contains API keys
- The `index/` folder must exist before starting the server — run `build_index.py` first
- `fwv_dataset.jsonl` is the fine-tuning dataset — not required for the chatbot to run
- `checkpoint.jsonl` is a dataset generation checkpoint — safe to delete after dataset is complete
